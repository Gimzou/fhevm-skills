# ERC-7984 Confidential Token

## Overview

ERC-7984 is a confidential fungible token standard where all balances and transfer amounts are encrypted using `euint64`. Unlike ERC-20:

- Balances are never visible on-chain (`confidentialBalanceOf` returns an encrypted handle, not a number)
- Transfers do not reveal the amount — the amount is encrypted throughout
- There are no numeric allowances; instead, **time-limited operators** grant delegated transfer rights until a block timestamp expires
- The standard is NOT ERC-20 compatible — there is no plaintext `balanceOf`, `transfer`, or `approve`

Balances are stored as `euint64` (6 decimals by default). All FHE operations use Zama's FHEVM coprocessor.

**`confidentialTransfer` returns the actual transferred amount** as a `euint64`. When a sender's balance is insufficient, `ERC7984._update` uses `FHESafeMath.tryDecrease`, which returns 0 instead of reverting. The return value must always be used — never assume the full requested amount was moved.

**Tooling requirements:** Solidity `^0.8.27` (the OpenZeppelin confidential-contracts base pins this pragma) and `@openzeppelin/contracts@^5.6.1` as a peer of `@openzeppelin/confidential-contracts`. Install with `--legacy-peer-deps`:
```bash
npm install --legacy-peer-deps \
  @fhevm/solidity@0.11.1 \
  @openzeppelin/confidential-contracts \
  @openzeppelin/contracts@^5.6.1
```

## Complete Example

See `templates/confidential-erc7984.sol` for the full compilable reference.

Key excerpt — concrete contract inherits both `ERC7984` and `ZamaEthereumConfig`:

```solidity
// @fhevm/solidity@0.11.1
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.27;

import { FHE, euint64, externalEuint64 } from "@fhevm/solidity/lib/FHE.sol";
import { ZamaEthereumConfig } from "@fhevm/solidity/config/ZamaConfig.sol";
import { ERC7984 } from "@openzeppelin/confidential-contracts/token/ERC7984/ERC7984.sol";

contract ConfidentialERC7984 is ZamaEthereumConfig, ERC7984 {
    // contractURI_ follows ERC-7572; pass "" for local/test deployments.
    constructor(string memory name_, string memory symbol_, string memory contractURI_)
        ZamaEthereumConfig()
        ERC7984(name_, symbol_, contractURI_)
    {}

    // Mint confidential tokens to msg.sender.
    // ERC7984._update handles allowThis + allow(balance, msg.sender) internally.
    function shield(externalEuint64 encryptedAmount, bytes calldata inputProof)
        external
        returns (euint64)
    {
        euint64 amount = FHE.fromExternal(encryptedAmount, inputProof);
        require(FHE.isSenderAllowed(amount), "Caller lacks ACL access");
        return _mint(msg.sender, amount);
    }

    // Burn encrypted tokens from msg.sender.
    // Caller must supply an on-chain euint64 handle they are ACL-allowed for.
    function unshield(euint64 amount) external returns (euint64) {
        require(FHE.isAllowed(amount, msg.sender), "Caller lacks ACL access");
        return _burn(msg.sender, amount);
    }
}
```

Hardhat test excerpt (`_validation/test/erc7984.ts`):

```typescript
// @fhevm/hardhat-plugin@0.4.2 | @zama-fhe/relayer-sdk@0.4.1
// Encrypt and shield 1000 tokens
const input = fhevm.createEncryptedInput(contractAddress, owner.address);
input.add64(1000n);
const { handles, inputProof } = await input.encrypt();
await token.shield(handles[0], inputProof);

// Decrypt owner balance
const balHandle = await token.confidentialBalanceOf(owner.address);
const balance = await fhevm.userDecryptEuint(FhevmType.euint64, balHandle, contractAddress, owner);
// balance === 1000n

// Transfer — disambiguate overloaded function by full signature
const transferFn = token["confidentialTransfer(address,bytes32,bytes)"];
const transferredHandle = await transferFn.staticCall(to, encHandle, proof);
await transferFn(to, encHandle, proof);
// Always use transferredHandle, not the requested amount
```

## API Reference

All functions are on the `ERC7984` abstract base contract at `@openzeppelin/confidential-contracts/token/ERC7984/ERC7984.sol`.

**Constructor (inherited — must call explicitly):**
- `constructor(string name_, string symbol_, string contractURI_)` — `contractURI_` follows ERC-7572; pass `""` for testing

**View functions:**
- `name() → string`
- `symbol() → string`
- `decimals() → uint8` — returns `6` by default
- `contractURI() → string`
- `confidentialTotalSupply() → euint64`
- `confidentialBalanceOf(address account) → euint64`
- `isOperator(address holder, address spender) → bool`

**Write functions:**
- `setOperator(address operator, uint48 until)` — grant delegated transfer rights until timestamp `until`
- `confidentialTransfer(address to, externalEuint64 encryptedAmount, bytes calldata inputProof) → euint64` — transfer with ZK proof; returns **actual** transferred amount
- `confidentialTransfer(address to, euint64 amount) → euint64` — transfer using on-chain handle (caller must be ACL-allowed); returns **actual** transferred amount
- `confidentialTransferFrom(address from, address to, externalEuint64 encryptedAmount, bytes calldata inputProof) → euint64` — delegated transfer; `msg.sender` must be operator for `from`
- `confidentialTransferFrom(address from, address to, euint64 amount) → euint64` — delegated transfer using on-chain handle

**Internal (for concrete contracts):**
- `_mint(address to, euint64 amount) → euint64 transferred` — ACL grants applied by `_update`
- `_burn(address from, euint64 amount) → euint64 transferred` — ACL grants applied by `_update`

> **Note:** `shield` and `unshield` shown in the Complete Example are **not** defined on the OpenZeppelin `ERC7984` base — they are concrete-contract wrappers around `_mint` / `_burn`. Define them (or equivalent domain-specific names) yourself when inheriting `ERC7984`.

**ERC7984ERC20Wrapper (two-step unwrap):**
- `wrap(address to, uint256 amount) → euint64` — lock ERC-20 tokens and mint confidential balance
- `unwrap(address from, address to, externalEuint64 encryptedAmount, bytes calldata inputProof) → bytes32 unwrapRequestId` — step 1: burn confidential tokens, make amount publicly decryptable, return request ID
- `finalizeUnwrap(bytes32 unwrapRequestId, uint64 unwrapAmountCleartext, bytes calldata decryptionProof)` — step 2: after Gateway delivers proof, release underlying ERC-20 tokens

**Ethers.js overload disambiguation** — `confidentialTransfer` is overloaded; always call it by full signature to avoid an `INVALID_ARGUMENT` error:
```typescript
// @fhevm/hardhat-plugin@0.4.2 | @zama-fhe/relayer-sdk@0.4.1
const tx = await token["confidentialTransfer(address,bytes32,bytes)"](to, handle, proof);
```

## Common Patterns

### Shield / Unshield cycle

```solidity
// @fhevm/solidity@0.11.1
// Shield: mint confidential balance
function shield(externalEuint64 encryptedAmount, bytes calldata inputProof) external returns (euint64) {
    euint64 amount = FHE.fromExternal(encryptedAmount, inputProof);
    require(FHE.isSenderAllowed(amount), "Caller lacks ACL access");
    return _mint(msg.sender, amount); // ACL grants handled by _update
}

// Unshield: burn confidential balance
function unshield(euint64 amount) external returns (euint64) {
    require(FHE.isAllowed(amount, msg.sender), "Caller lacks ACL access");
    return _burn(msg.sender, amount);
}
```

### confidentialTransfer with actual-amount guard

```solidity
// @fhevm/solidity@0.11.1
// The actual transferred amount may be less than requested (insufficient balance).
// ERC7984._update uses FHESafeMath.tryDecrease, which returns 0 on underflow.
// Always use the return value of confidentialTransfer — never the input amount.
euint64 actualTransferred = token.confidentialTransfer(to, encryptedAmount, inputProof);
_processActualAmount(actualTransferred); // e.g. update accounting, emit event
```

### Operator approval pattern (time-bounded)

```solidity
// @fhevm/solidity@0.11.1
// Grant operator rights to `spender` until a future timestamp.
// Unlike ERC-20 allowances, there is no encrypted amount limit — the operator
// may move any amount until `until` expires.
uint48 expiry = uint48(block.timestamp + 7 days);
token.setOperator(spenderAddress, expiry);

// Operator executes delegated transfer:
token.confidentialTransferFrom(holder, recipient, encryptedAmount, inputProof);
```

### ERC7984ERC20Wrapper two-step unwrap

```solidity
// @fhevm/solidity@0.11.1
// Step 1 — initiate unwrap: burns confidential tokens, requests KMS decryption.
// requestId is the euint64 handle of unwrapAmount_, cast to bytes32.
bytes32 requestId = wrapper.unwrap(msg.sender, recipientAddress, encryptedAmount, inputProof);
// The contract emits UnwrapRequested(to, requestId, unwrapAmount_)
// Gateway/KMS decrypts unwrapAmount_ and calls back with a proof

// Step 2 — finalize: called after Gateway delivers the decryption proof.
// finalizeUnwrap calls FHE.checkSignatures internally — a valid KMS-signed proof is mandatory.
wrapper.finalizeUnwrap(requestId, cleartextAmount, decryptionProof);
// Releases cleartextAmount * rate() underlying ERC-20 tokens to recipient.
```

**Rate and decimals.** `ERC7984ERC20Wrapper` caps confidential `decimals()` at 6. When the underlying ERC-20 uses more (e.g. 18), `rate() = 10**(underlyingDecimals - 6)` and `wrap(...)` rounds the deposited amount down to the nearest multiple of `rate()` (any remainder is refunded inline). Unwrapping `N` confidential units releases `N * rate()` underlying tokens.

**Testing limitation — `finalizeUnwrap` in mock mode.** The Hardhat mock provider bundled with `@fhevm/hardhat-plugin@0.4.2` does not expose the `fhevm_createDecryptionSignatures` RPC that `FHE.checkSignatures` requires. Tests can validate `wrap()` and the first `unwrap()` step (confidential balance decreases, `UnwrapRequested` is emitted, `unwrapRequester(requestId)` returns the target), but cannot exercise `finalizeUnwrap` end-to-end. Full coverage requires Sepolia + the Gateway callback.

## Anti-Patterns

**❌ Ignoring the return value of `confidentialTransfer`**
> `confidentialTransfer` returns the **actual** transferred amount as `euint64`. When the sender's balance is insufficient, `ERC7984._update` saturates at zero via `FHESafeMath.tryDecrease` — no revert. Using the requested amount instead of the return value silently corrupts downstream accounting.
> ```solidity
> // @fhevm/solidity@0.11.1
> // ❌ Assumes full amount was transferred
> _burn(from, requestedAmount);
> ```

**✅ Correct: always use the returned amount**
> ```solidity
> // @fhevm/solidity@0.11.1
> euint64 actualTransferred = token.confidentialTransfer(to, amount, inputProof);
> _processActualAmount(actualTransferred);
> ```

---

**❌ Wrong package name — `@fhevm/sdk` does not exist on npm**
> Importing from `@fhevm/sdk` produces a 404 error. The correct package for ERC-7984 token operations is `@zama-fhe/sdk@2.2.0`.
> ```typescript
> // ❌ 404 on npm
> import { ZamaSDK } from "@fhevm/sdk";
> ```

**✅ Correct package name**
> ```typescript
> // @zama-fhe/sdk@2.2.0 | @zama-fhe/relayer-sdk@0.4.1
> import { ZamaSDK } from "@zama-fhe/sdk";
> ```

---

**❌ Using `fhevmjs` in new code**
> `fhevmjs` is a legacy client-side SDK. For new ERC-7984 token integrations, use `@zama-fhe/sdk`. For arbitrary custom contracts (voting, gaming, etc.), use `@zama-fhe/relayer-sdk` directly.
> ```typescript
> // ❌ Legacy — do not use in new code
> import { createInstance } from "fhevmjs";
> ```

---

**❌ Missing `ZamaEthereumConfig` when inheriting `ERC7984`**
> `ERC7984` inherits from `ERC165` only — it does NOT register the FHE coprocessor. Omitting `ZamaEthereumConfig` from your concrete contract will cause FHE operations to fail on mainnet and Sepolia (reverts with `ZamaProtocolUnsupported()` or incorrect coprocessor behavior).
> ```solidity
> // @fhevm/solidity@0.11.1
> // ❌ Missing coprocessor registration
> contract MyToken is ERC7984 { ... }
> ```

**✅ Correct — inherit both**
> ```solidity
> // @fhevm/solidity@0.11.1
> contract MyToken is ERC7984, ZamaEthereumConfig {
>     constructor(string memory name_, string memory symbol_, string memory contractURI_)
>         ERC7984(name_, symbol_, contractURI_)
>         ZamaEthereumConfig()
>     {}
> }
> ```
